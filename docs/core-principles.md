# 核心原则

本文档详细阐述 Harness Engineering 的核心原则。

## 原则 1: 双向控制闭环

### 定义

系统必须同时部署引导指令和检测传感器：

```
┌─────────────────────────────────────────────────────────────┐
│                     双 向 控 制 闭 环                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌───────────────┐        ┌───────────────┐               │
│   │   引导指令      │──────▶│   Agent 执行   │               │
│   │ (行动前干预)    │        │               │               │
│   └───────────────┘        └───────┬───────┘               │
│           ▲                       │                       │
│           │                       ▼                       │
│           │              ┌───────────────┐                │
│           └──────────────│   检测传感器   │                │
│                (行动后观察)│   (反馈控制)  │                │
│                           └───────────────┘                │
└─────────────────────────────────────────────────────────────┘
```

### 引导指令类型

| 类型 | 用途 | 示例 |
|------|------|------|
| 规范文件 | 定义代码风格 | `AGENTS.md`, `.editorconfig` |
| 模板 | 预置代码结构 | `template.ts`, `component.vue` |
| 约束 | 禁止/强制行为 | ESLint 规则, `.gitignore` |
| 检查点 | 分阶段验证 | CI/CD gate |

### 检测传感器类型

| 类型 | 用途 | 示例 |
|------|------|------|
| 静态分析 | 代码质量 | ESLint, Prettier |
| 测试验证 | 功能正确性 | Jest, Pytest |
| 性能监控 | 运行时指标 | APM, 基准测试 |
| AI 审查 | 语义评估 | Code Review Agent |

### 缺失任一环节的后果

- **只有引导指令**: Agent 执行但无法纠错，重复犯错
- **只有检测传感器**: Agent 盲目执行，事后才知错误

## 原则 2: 混合执行模式

### 概念

结合计算型控制和推断型控制：

```
┌─────────────────────────────────────────────────────────────┐
│                      混 合 执 行 模 式                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │              计算型控制 (Compute)                     │  │
│   │  - 快速执行                                           │  │
│   │  - 确定性强                                           │  │
│   │  - 低成本                                            │  │
│   │  - 示例: Linter, Type Check, Regex                   │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │              推断型控制 (Inference)                   │  │
│   │  - 语义理解                                           │  │
│   │  - 上下文感知                                         │  │
│   │  - 成本较高                                          │  │
│   │  - 示例: AI Code Review, Semantic Analysis          │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 分层策略

```yaml
execution_layers:
  - name: "快速检查 (Pre-commit)"
    type: compute
    tools: [eslint, prettier, typescript]
    trigger: on_save

  - name: "CI 检查 (On Push)"
    type: compute
    tools: [test, build, lint]
    trigger: on_push

  - name: "深度评估 (On PR)"
    type: hybrid
    tools: [coverage, complexity, ai-review]
    trigger: on_pr

  - name: "AI 审查 (Async)"
    type: inference
    tools: [code-review-agent, semantic-check]
    trigger: scheduled
```

### 选择指南

| 场景 | 推荐类型 | 原因 |
|------|----------|------|
| 代码格式 | 计算型 | 确定性、无需理解语义 |
| 类型检查 | 计算型 | 可自动化、无歧义 |
| 安全漏洞 | 混合型 | 需要模式匹配+语义理解 |
| 逻辑错误 | 推断型 | 需要上下文理解 |
| 性能问题 | 混合型 | 数据测量+模式识别 |

## 原则 3: 质量前置策略

### 核心理念

将验证步骤按成本与时效分层，越早发现问题越好：

```
成本
  ▲
  │                              ┌─────────────┐
  │                              │ 深度评估    │ ← 集成流水线
  │                         ┌────┤  (CI/CD)    │
  │                         │    └─────────────┘
  │                    ┌────┤
  │                    │    │    ┌─────────────┐
  │               ┌────┤    │    │ 高频快检    │ ← Pre-commit
  │               │    │    └────┤  (Local)    │
  │               │    │         └─────────────┘
  │          ┌────┤    │
  │          │    │    │         ┌─────────────┐
  └──────────┴────┴────┴─────────┤ 持续监控    │ ← Scheduled
                │                └─────────────┘
                └──────────────── 时间 ─────────────────────▶
```

### 验证分层

| 层级 | 时机 | 内容 | 工具 |
|------|------|------|------|
| L1 零级 | 编辑时 | 实时语法检查 | Language Server |
| L2 一级 | 保存时 | 格式化、Lint | Pre-commit |
| L3 二级 | 提交时 | 单元测试 | Git Hooks |
| L4 三级 | 推送时 | 构建、集成测试 | CI/CD |
| L5 四级 | 合并时 | 代码审查、深度测试 | PR Workflow |
| L6 五级 | 部署后 | 监控、告警 | APM |

### 瀑布式反馈

```
问题发现 ──────▶ 修复成本
  │               │
  │    ┌──────────┴──────────┐
  ▼    ▼                     ▼
 L1   L3                     L5
 1x   10x                   100x
```

越早发现问题，修复成本越低。

## 原则 4: 多样性约束定律

### 控制论基础

根据控制论原理，调节器需具备对应系统的复杂度：

```
┌─────────────────────────────────────────────────────────────┐
│                   复 杂 度 匹 配 原 则                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   系统复杂度 ─────────────────────────────▶                 │
│      ▲                                                    │
│      │                                                    │
│      │      ┌─────────────────────────────────┐          │
│      │      │                                 │          │
│      │      │   如果调节器复杂度 < 系统复杂度   │          │
│      │      │   则无法有效控制系统             │          │
│      │      │                                 │          │
│      │      └─────────────────────────────────┘          │
│      │                                                    │
│      │      ┌─────────────────────────────────┐          │
│      │      │                                 │          │
│      │      │   如果调节器复杂度 > 系统复杂度   │          │
│      │      │   则产生过度约束，降低效率        │          │
│      │      │                                 │          │
│      │      └─────────────────────────────────┘          │
│      │                                                    │
│      └──────────────── 调节器复杂度                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 约束策略

**目标**: 通过固化标准架构拓扑来压缩 Agent 可能生成的代码范围。

### 具体方法

1. **标准拓扑**: 强制执行统一的架构模式
2. **边界清晰**: 定义明确的模块边界和接口
3. **模式库**: 提供经过验证的代码模式
4. **黑名单/白名单**: 明确允许和禁止的技术

### 约束强度

| 强度 | 类型 | 示例 | Agent 灵活性 |
|------|------|------|--------------|
| 强制 | 必须遵守 | 禁止 `eval()` | 无 |
| 强制 | 必须遵守 | 依赖白名单 | 极低 |
| 指导 | 推荐遵循 | 命名规范 | 中等 |
| 建议 | 可选择 | 代码风格 | 高 |

## 实践指南

### 设计约束的步骤

1. **识别系统复杂度**
   - 模块数量
   - 依赖关系
   - 变更频率

2. **确定监管需求**
   - 关键质量属性
   - 可接受的风险水平
   - 合规要求

3. **设计调节器**
   - 选择控制类型（计算/推断）
   - 决定约束强度
   - 规划反馈机制

4. **验证有效性**
   - 模拟 Agent 行为
   - 测试边界情况
   - 测量监管覆盖率

### 常见错误

- **过度约束**: 限制 Agent 灵活性，导致无法完成任务
- **约束不足**: Agent 行为无法预测，产生质量问题
- **静态约束**: 不随系统演进而更新
- **碎片化约束**: 多个约束冲突，无统一策略
