# 最佳实践

本文档汇总 Harness Engineering 的最佳实践和经验总结。

## 设计阶段最佳实践

### 1. 从简单开始

> **"构建可撕裂的 Harness"** —— 避免过度工程化

```
阶段1: 简单 Harness
┌─────────────────────────────────────────────────────────────┐
│  - Convention 文件                                          │
│  - Pre-commit hooks                                        │
│  - 基础 Lint 配置                                          │
└─────────────────────────────────────────────────────────────┘

阶段2: 中等 Harness
┌─────────────────────────────────────────────────────────────┐
│  + CI/CD 集成                                              │
│  + 自动化测试                                               │
│  + 基础监控                                                 │
└─────────────────────────────────────────────────────────────┘

阶段3: 完整 Harness
┌─────────────────────────────────────────────────────────────┐
│  + 多 Agent 编排                                           │
│  + 深度评估                                                 │
│  + 全面可观测性                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 渐进式约束

| 阶段 | 约束类型 | 强度 | Agent 灵活性 |
|------|----------|------|--------------|
| 初期 | 软约束 | 低 | 高 |
| 稳定 | 硬约束 | 中 | 中 |
| 生产 | 强制 | 高 | 低 |

## 开发阶段最佳实践

### 1. 上下文管理

#### 正确的上下文注入

```yaml
# ✅ 正确: 有序注入
context_injection:
  order:
    - system_overview      # 系统概述
    - architecture_rules   # 架构规则
    - coding_standards     # 编码标准
    - task_specific        # 任务特定

# ❌ 错误: 无序堆砌
context_injection:
  order:
    - task_specific        # 任务特定
    - system_overview      # 系统概述
    - task_specific        # 重复
    - architecture_rules   # 架构规则
```

#### 上下文预算

```
Token 预算分配建议:
├── 系统指令: 10-15%
├── 架构规则: 15-20%
├── 任务描述: 10-15%
├── 相关代码: 40-50%
└── 对话历史: 10-15%
```

### 3. 工具设计

#### 清晰的工具边界

```yaml
# ✅ 好: 明确的工具定义
tools:
  read_file:
    description: "读取单个文件内容"
    parameters:
      path:
        type: string
        required: true
        pattern: "^src/.*"
    returns:
      type: object
      properties:
        content: string
        lines: number

# ❌ 差: 模糊的工具定义
tools:
  read:
    description: "读取内容"
    parameters:
      data:
        type: any
```

#### 工具组合

```
简单任务 ──▶ 单工具
         ──▶ read_file

中等任务 ──▶ 工具链
         ──▶ read_file ──▶ grep ──▶ read_file

复杂任务 ──▶ 工具编排
         ──▶ search ──▶ read_many ──▶ analyze ──▶ write ──▶ test
```

## 常见错误与解决方案

### 1. 上下文溢出

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Agent 行为不稳定 | 上下文太长 | 实施上下文压缩 |
| 响应质量下降 | 关键信息被截断 | 优先级排序 |
| Token 成本上升 | 无效信息过多 | 清理冗余内容 |

### 2. 反馈延迟

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Agent 重复犯错 | 反馈太慢 | 增加快速检查 |
| 任务失败率高 | 错误未及时发现 | 前置验证 |
| 开发体验差 | 等待时间长 | 异步反馈 |

### 3. 约束冲突

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Agent 无所适从 | 多套规则冲突 | 统一规则体系 |
| 任务无法完成 | 约束过于严格 | 优先级分层 |
| 规则过时 | 未更新 | 定期审查 |
